# Prompts

Two short, focused prompts. One agent per prompt, running in parallel.

| Prompt | Agent | Owns |
|---|---|---|
| [implementation.md](implementation.md) | implementation agent | `crates/**/src/*.rs` and the `@parity` tags in them (drives `source-map.md`) |
| [test-parity.md](test-parity.md)       | test parity agent    | `crates/**/tests/*.rs` and `mod tests` blocks, `// Ported:` comments (drives `test-map.md`) |

They do **not** edit the same files. Both tables are generated by `parity-cli`,
not typed by hand. Each agent works **one row at a time** — the implementation
agent one upstream `.ts` file, the test parity agent one upstream spec.

## How priorities are set

`docs/parity/milestones.md` lists the ordered milestones. Both agents always
pick work from the **first incomplete milestone**.

## Tools

`parity-cli` (Rust) owns the two file-level parity maps and writes each to its
own location automatically — no output redirection. It scans the whole upstream
repo for inputs, so new / moved / removed files are picked up on every run:

```sh
# Run anywhere in renovate-rust/
cargo run -p parity-cli                # regenerate BOTH maps
cargo run -p parity-cli -- source      # docs/parity/source-map.md (TS impl → Rust, @parity tags)
cargo run -p parity-cli -- test        # docs/parity/test-map.md   (upstream it() → // Ported:)
cargo run -p parity-cli -- check       # CI guard: stale @parity tags + deleted-upstream tests
cargo run -p parity-cli -- normalize   # one-time: canonicalize all // Ported: refs to lib/... form
```

- **source-map** rows: `full` · `partial` · `stub` · `pending` (no `@parity`
  tag) · `out-of-scope`. The implementation agent sets status via `@parity`
  tags in the Rust source.
- **test-map** states: `ported` (upstream test + matching `// Ported:`),
  `pending` (upstream test, no port), `deleted` (a `// Ported:` whose upstream
  identity is gone — kept for review, never auto-removed).

Per-file lookups are intentionally absent — `grep docs/parity/source-map.md`
(or `test-map.md`) answers them against the generated table.

The Python `scripts/parity_coverage.py` remains the older **module-level**
ledger (`modules.md`); it is complementary but the per-file `parity-cli` maps
are the primary surface the prompts drive.

## Invocations

Run both agents from the workspace root `~/Projects/renovate-rust-experiement`.

### Implementation agent

```text
Follow @renovate-rust/prompts/implementation.md. Regenerate the source map,
then work the first incomplete milestone in @renovate-rust/docs/parity/
milestones.md: pick one pending/partial row, implement it, set its @parity tag,
regenerate, and run `parity-cli -- check`. Commit each coherent slice with the
required Co-authored-by trailer and push.
```

### Test parity agent

```text
Follow @renovate-rust/prompts/test-parity.md. Regenerate the test map, then work
the first incomplete milestone in @renovate-rust/docs/parity/milestones.md: pick
one spec with pending tests, port them with canonical // Ported: comments,
regenerate, and run `parity-cli -- check`. Skip specs whose implementation is
still pending. Commit each batch with the required Co-authored-by trailer and push.
```

For non-interactive Codex, replace `@renovate-rust/prompts/...` with
`$(cat renovate-rust/prompts/implementation.md)` etc.

## Prompt maintenance

The two prompts are operator-owned. Agents running them must not edit them.
File improvement suggestions in `docs/parity/prompt-improvements.md` — the
operator decides whether to apply.
